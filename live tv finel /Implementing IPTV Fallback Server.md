## 🎯  Root‑## 🎯  Root‑Cause Analysis (Why the **`rejected` / `push failed`** error keeps coming back)

| Step | What actually happens in the GitHub Action | Why it breaks |
|------|--------------------------------------------|---------------|
| **1️⃣ Checkout** | `actions/checkout@v4` does a **shallow clone** (`fetch‑depth: 1` by default). Only the latest commit is present locally. | The local repo **doesn’t have the full history** of `channels.json`. When we later try to re‑base or push, Git can’t find the upstream base commit. |
| **2️⃣ Run the checker script** | `npm start` generates a **new `channels.json`** (tens of thousands of lines). This file is **different** from the one that exists on the remote. | The Action now has a *new* version of the file that the remote repository doesn’t know about. |
| **3️⃣ Commit & Push** | The workflow runs `git add channels.json && git commit … && git pull --rebase && git push`. Because the checkout was shallow, the `git pull --rebase` **fails** (no common ancestor) and the subsequent `git push` is rejected with the classic message: <br>`! [rejected] main -> main (fetch first)` | Git tells us “**you need to fetch the latest remote changes before you can push**”. The shallow checkout prevents a proper fetch/rebase, so the push is always rejected. |
| **4️⃣ Re‑run the workflow** | The same steps repeat → the same error appears **every time**. | Nothing in the workflow changes the underlying problem, so the error is deterministic. |

### Secondary code‑side issue (still visible in the UI)

Even after the Action succeeds, the **client‑side UI** still shows many channels as **offline** because:

* The app trusts the **GitHub‑provided health status** (`channelHealth[channel.link]`) which is computed on GitHub’s US/EU servers.  
* Many Indian IPTV streams are **geo‑blocked** – GitHub sees them as *offline* and marks them red.  
* The app then **auto‑skips** Server 1 based on that false status, causing playback failures.

| Change | Reason |
|--------|--------|
| **a. Full clone** – `fetch-depth: 0` | Gives the runner the complete history so `git pull --rebase` can find a common ancestor. |
| **b. Use `git-auto-commit-action`** (or our own safe push script) | Handles “no‑changes” case automatically and does the commit/push in one atomic step. |
| **c. Add a **concurrency group** (`concurrency: iptv-checker`)** | Prevents two runs from overlapping and fighting over the same file. |
| **d. Force‑with‑lease** (`git push --force-with-lease`) | Guarantees we only overwrite our own commit, never someone else’s work. |
